SECTION · 01 · PROBLEM

문제

학내 50여 동아리가 공용 회의실을 수기 장부와 단톡방으로 예약해 왔습니다. 같은 시간에 두세 팀이 들어오는 겹침 사고가 매주 반복되고, 노쇼는 다음 팀 활동을 끊어버렸습니다.

"누가, 언제, 어디에 있는지 아무도 정확히 모른다."

SECTION · 02 · RESEARCH

리서치

회장단을 대상으로 구글폼 설문을 돌려 페인포인트를 정렬했습니다. 가장 강한 시그널은 '실시간으로 누가 어디를 쓰는지 보고 싶다'는 니즈였고, 기존 장부·단톡 방식의 비동기성이 겹침과 노쇼의 근원이었습니다.

  • 회장단 설문으로 니즈 우선순위 도출
  • 장부·단톡 기록과 사고 사례 교차 확인
  • '실시간 현황 확인'을 최우선 가치로 확정
— 회장단 설문 리뷰
SECTION · 03 · MVP DECISION

MVP 결정

백엔드 2명·한 학기 일정이라는 제약 안에서 무엇을 빼고 무엇을 남길지가 핵심이었습니다. '실시간 현황 확인'이라는 단 하나의 가치를 1차 범위로 정하고, 알림은 2차로 미뤘습니다.

01

예약 · 취소

회의실/시간 슬롯을 선택해 등록하고 본인이 취소 가능.

02

실시간 뷰

전체 회의실의 현재/오늘 점유 상태를 한 화면에서 확인.

03

알림 (2차)

리마인드·노쇼 경고는 다음 스프린트로 분리, MVP에서 제외.

— 빼는 결정이 만드는 결정보다 어려웠다.

SECTION · 04 · EXECUTION

실행

백엔드와의 커뮤니케이션 비용을 줄이기 위해 API 명세 초안을 직접 작성해 전달했습니다. 엔드포인트, 요청/응답 스키마, 에러 케이스를 미리 합의하니 구현 단계에서 되돌아오는 질문이 줄었습니다.

Jira로 백로그를 쪼개고 1주 스프린트 단위로 데모를 돌렸습니다. 회장단 두세 명을 매주 리뷰에 초대해 그 자리에서 우선순위를 갱신했습니다.

  • API 명세 초안 직접 작성 → 합의 시간 단축
  • Jira · 스크럼 · 주 1회 데모로 피드백 루프 단축
  • 회장단 상시 리뷰로 범위 변경을 빠르게 흡수
SECTION · 05 · RESULT

결과

30 동아리 · −60% 노쇼
— 출시 1개월 시점

출시 1개월 만에 학내 30개 동아리가 도입했고, 노쇼는 60% 감소했습니다. '회의실 단톡 알림'이 사라진 자리에 실시간 뷰가 들어왔습니다.

"끊어진 흐름이 다시 이어졌다."

다음 분기에 알림·리마인드·관리자 통계 모듈을 2차로 붙이며 운영을 회장단에 이관했습니다. PM이 빠진 뒤에도 유지되는 구조를 만드는 것이 마지막 책임이었습니다.
