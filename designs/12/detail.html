CASE STUDY: ROOMSYNC

학내 동아리방 예약 서비스 — 30개 동아리 / 노쇼 -60%
기간2023.07 — 2023.11
역할서비스 기획 / PM (요구사항 정의, API 명세 초안, 스프린트 관리)
팀기획 1명 (본인) + 백엔드 2명
도구Notion, Figma, Jira, SQL, Google Forms
결과30개 동아리 가입, 노쇼율 -60%
SECTION 01

문제 / Problem

학내 50여 개 동아리가 공용 회의실을 수기 장부와 카톡으로 예약하면서, 같은 시간대 중복 예약과 잠금 사고가 반복되었습니다. 회장단은 "지금 누가 쓰고 있는지"를 묻는 메시지에 반복적으로 응대해야 했고, 신청자는 직접 방으로 가서야 점유 여부를 확인할 수 있었습니다.

  • 중복 예약 / 잠금 사고 반복
  • 회장단 실시간 문의 응대 부담
  • 예약 후 미사용(노쇼) 체감 증가
SECTION 02

리서치 / Research

회장단 대상 구글폼 설문을 진행해 페인포인트와 우선순위를 수집했습니다. 응답을 정리한 결과, "지금 누가 쓰고 있는지"를 즉시 확인할 수 있는 실시간 현황이 가장 빈번하게 언급된 니즈로 드러났습니다.

[ PHOTO #03 / USER_INTERVIEW.JPG / 2023 ]
  • 구글폼 설문: 회장단 대상
  • 최우선 니즈로 실시간 현황 확인 도출
  • 알림은 후순위로 분류 → v2로 deferred
  • 예약 자체보다 점유 가시성이 결정적 페인포인트라는 결론
SECTION 03

MVP 결정 / Decision

"전부 다 만들고 싶다"를 누르고, MVP에는 핵심 기능만 담았습니다. 알림은 2차로 명시적으로 deferred 처리해 백엔드 2명의 공수를 핵심 흐름에 집중시켰습니다.

  • ① 예약 / 취소
  • ② 실시간 점유 뷰
  • ③ 회장단 어드민 (강제 취소 권한)
  • v2 deferred: 알림

API 명세 초안은 본인이 직접 작성해 백엔드와 합의 시간을 줄였고, 엣지 케이스(동시 클릭, 권한 미스매치)는 Notion에 시나리오로 문서화했습니다.

SECTION 04

실행 / Execution

[ PHOTO #01 / PM_PORTRAIT.JPG / 2023 ]

Jira 기반 스크럼으로 운영하며, 백엔드 2명과 함께 핵심 흐름부터 점진적으로 구현했습니다. PM이 API 명세 초안을 먼저 작성해 합의 시간을 줄였습니다.

  • Jira·스크럼 스프린트 운영
  • API 명세 초안 직접 작성
  • 단계적 베타 → 정식 확대
SECTION 05

결과 / Result

[ PHOTO #04 / LECTURE_DEMODAY.JPG / 2023 ]

가입 동아리 30개, 노쇼율은 -60%를 기록했습니다. "지금 누가 쓰고 있는지"를 데이터로 보여주는 단 하나의 화면이, 회장단 응대 부담과 미사용 문제를 동시에 줄였습니다.

  • 30개 동아리 가입
  • 노쇼율 -60%
  • v2 로드맵: 알림
