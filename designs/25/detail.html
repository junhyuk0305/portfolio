Research — 회장단의 목소리

설문 한 장이 범위를 잘랐다.

Photo by 김태윤 · 2023 회장단 인터뷰. 한 줄의 합의가 ‘만들 것’과 ‘덜어낼 것’을 갈랐다.

RESEARCH · 회장단 구글폼 설문 · 현장 관찰

회장단을 대상으로 구글폼 설문을 돌렸습니다. 가장 큰 응답은 의외였습니다. 사람들은 ‘예약 기능’이 아니라 ‘지금 방이 비었는가’를 알고 싶어 했습니다. 가장 큰 페인 포인트는 ‘실시간 현황 확인’이었습니다.

이 한 줄이 제품 범위의 칼이 되었습니다. ‘예약’은 당연한 코어, ‘취소’는 회복 동선, ‘실시간 뷰’는 차별점. 알림은 이번 사이클의 대상이 아니라고 결정했습니다.

핵심 인사이트

  • 충돌의 원인은 정보 부재였다 — 예약 부재가 아니라.
  • ‘지금’이라는 시간 축이 모든 화면의 중심에 있어야 했다.
  • 알림은 좋지만, 한 달 안에 배포할 수는 없었다.
“만들 것을 정하는 일보다, 만들지 않을 것을 정하는 일이 더 어려웠다.” — PM 노트
Photo by 김태윤 · 2023 결정의 순간. ‘덜어냄’이 곧 일정이었다.
DECISION · 3 of 5
한 달이라는 마감이 설계를 결정했다.
PM이 직접 API 명세 초안을 잡았다. 백엔드 두 명이 동시에 다른 도메인을 잡아당기게 하려면, 인터페이스가 회의보다 먼저 와야 했다.

MVP 범위 (1차)

  • 예약 — 시간대 잡기, 충돌 시 즉시 거절
  • 취소 — 1탭 취소, 회복 가능한 동선
  • 실시간 뷰 — 지금 비어 있는 방의 한눈 보기

2차로 미룬 것

  • 알림 — 한 달 안에 안정성을 보장할 수 없음

스크럼은 1주 단위로 돌렸고, Jira로 백엔드 두 명의 작업을 같은 페이지에 묶었습니다. PM이 그린 API 명세 초안은 매주의 정렬점이었고, ‘무엇을 만들고 있는가’를 회의 없이도 확인할 수 있게 해주었습니다.

PAGE 064 · 실행
Execution — 한 달의 사이클

회의가 아니라 명세가 먼저였다.

Photo by 김태윤 · 2023 스크럼 보드. 같은 페이지 위에서 두 백엔드가 평행하게 달렸다.

1주차에 도메인을 쪼개고, 2주차에 API 명세 초안과 와이어를 동시에 굳혔습니다. 3주차에 핵심 동선을 끝내고, 4주차는 실데이터로 30개 동아리 회장단과 함께 베타를 돌렸습니다.

막힌 곳은 ‘충돌 정책’이었습니다. 동시에 같은 시간을 잡은 두 요청을 어떻게 처리할 것인가 — 단순한 ‘선착순’이 사용자에게 ‘공정함’으로 느껴지도록 메시지와 타임스탬프를 함께 노출했습니다. UX의 문제가 정책의 문제임을, 이때 배웠습니다.

운영 도구

Jira · Scrum (1주 사이클) · Figma · API 명세 초안 · 회장단 베타

“API 명세는 디자인보다 먼저, 회의보다 먼저였다.” — PM 노트

한 달의 MVP 끝에, 30개 동아리가 들어왔고 노쇼는 60% 감소했습니다. 더 많은 동아리를 받기 위해서는 ‘알림’이 다음 마일스톤이 되어야 함을 데이터가 말해주었습니다.

이 한 달이 가르쳐 준 것은 단순합니다 — 좋은 기획은 ‘더 많이 그리는 일’이 아니라 ‘무엇을 덜어낼지 정하는 일’입니다.
