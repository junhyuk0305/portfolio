CASE  01
— ROOMSYNC

학내 동아리방 예약 웹서비스 2023.07 – 2023.11 PM · 5mo · 백엔드 2명
[FIG.01] 학내 동아리방, 2023

문제/ where it hurts

수기 예약 → 겹침 사고 잦음

학내에는 50여 개 동아리가 공용 회의실 몇 개를 함께 씁니다. 예약은 오프라인 노트와 카톡으로 이뤄졌고, 같은 시간대에 두 동아리가 회의실 앞에서 마주치는 겹침 사고가 매주 반복됐습니다.

회장단을 대상으로 구글폼 설문을 돌렸더니, 가장 큰 니즈는 “지금 이 방이 비었는지 실시간으로 확인하고 싶다” 였습니다.

“결국 회의실 앞에 가봐야 비었는지 알 수 있어요.” — 회장단 설문, 2023.07

가설

  • 회장단이 진짜 원하는 건 실시간 현황 확인이다.
  • 알림 · 통계는 2차로 미루고 예약 · 취소 · 실시간 뷰만 먼저 만들자.
  • 겹침이 사라지면 노쇼율도 함께 떨어질 것이다.

리서치/ listen first

📌 회장단 설문, 2023.07

회장단 설문 + 사용 패턴 관찰

50여 개 동아리 회장단을 대상으로 구글폼 설문을 돌리고, 공용 회의실 앞에서 일주일간 실제 사용 패턴을 직접 관찰했습니다. 예약은 종이 노트와 회장단 단톡방에 흩어져 있었습니다.

  • 1순위 니즈: ‘실시간으로 빈 방 확인’ (응답의 다수)
  • 겹침 사고의 대부분은 예약 변경이 단톡방에만 공유됐기 때문
  • 알림 · 통계 같은 기능은 “있으면 좋고”에 머묾
“지금 비었는지만 알아도 절반은 해결돼요.” — 회장단 설문 자유응답

실행/ MVP 전략

MVP 전략 + API 명세 초안

백엔드 2명과 함께 일정을 짜며 ‘예약 · 취소 · 실시간 뷰’ 세 가지 핵심 기능만 1차 범위로 묶었습니다. 푸시 알림 · 사용 통계 등 부가 기능은 2차로 명시적으로 미뤘습니다. 제가 API 명세 초안을 먼저 그려 백엔드와 싱크를 맞췄고, 스크럼 마스터 역할로 Jira에 스프린트를 끊었습니다.

  • 1차 범위: 예약 / 취소 / 실시간 현황 뷰
  • 2차 범위: 푸시 알림 · 통계 대시보드 (의도적으로 컷)
  • API 명세 초안 직접 작성 → 백엔드와 매일 싱크
  • Jira로 2주 단위 스프린트 · 데일리 15분 스탠드업
[FIG.02] 스프린트 리뷰

역할

  • PM · 스크럼 마스터
  • 요구사항 정리 → PRD → API 명세 초안
  • 회장단 베타테스트 운영 (3주)
  • 출시 후 30일 회고 작성

결과/ numbers don't lie ✿

📌 출시 1개월 회고, 2023.12
30개 동아리 가입 · 노쇼 -60%

출시 1개월

베타 종료 후 정식 오픈, 한 달 만에 50개 중 30개 동아리가 가입했습니다. 가장 큰 변화는 겹침 사고가 사실상 사라지면서 노쇼율이 60% 감소한 점. “지금 비었나?”라는 단톡방 질문이 줄었다는 정성 피드백이 가장 인상 깊었습니다.

배운 것/ note to self

1. 범위를 줄이는 게 진짜 일

“알림도, 통계도, 길찾기도 다 있으면 좋겠다”는 말을 들었지만, 백엔드 2명 · 5개월이라는 제약 안에서 ‘예약 · 취소 · 실시간 뷰’만 남기는 결정이 가장 어렵고도 중요했습니다.

2. PM이 직접 그려야 빨라진다

API 명세 초안을 PM이 먼저 그려서 들고 가니, 백엔드와의 싱크 회의가 절반으로 줄었습니다. 완벽하진 않아도, 대화의 출발점이 되는 문서가 있다는 게 중요했습니다.

3. PM이 가장 자주 한 말

“이 기능, 지금 꼭 필요한가요?
2차로 미루면 안 될까요?” — 회고 노트, 2023.11

CASE 02/ A사 커머스 인턴 · 2023.01–06

장바구니에서 45% 이탈 발견

MAU 50만 서비스에서 GA4 · 앰플리튜드로 퍼널을 분석했더니, 장바구니 → 결제 단계 사이에 이탈 45%가 누적되고 있었습니다. 숫자만으로는 이유를 알 수 없어 5명 UT를 돌렸고, 하단 “구매하기” 버튼이 모바일에서 키보드/홈바에 가려진다는 공통 행동을 관찰했습니다.

  • 가설: 버튼이 시야에서 사라지는 ‘찰나의 망설임’이 이탈을 만든다
  • 실험: 버튼 투명도·높이 조정 + “무료배송 N원 남음” 넛지를 A/B
  • 결과: B안 전환율 +12%p → 정규 배포 확정
“버튼이 안 보이니까 일단 닫고, 다시 안 들어왔어요.” — UT P02

역할 / 도구

  • 인턴 PM (사수와 페어)
  • GA4 · Amplitude · SQL
  • UT 모더레이션 5세션
  • A/B 실험 설계 보조

CASE 03/ IT 연합 동아리 회장 · 2022.03–12

완성률 40 → 85%로 끌어올린 운영

회원 100명 규모 IT 연합 동아리의 회장을 맡았습니다. 가장 큰 문제는 학기 초에 모인 프로젝트팀이 중반에 흩어지고, 실제 완성 비율이 40%에 그친다는 점이었습니다.

  • 크로스오버 해커톤: 기획자–개발자–디자이너를 강제 매칭, 36h 단기 산출물
  • ‘서로의 언어’ 세미나: 직군 간 용어를 서로 가르치는 격주 세션
  • 결과: 학기 말 완성률 85%, 다음 해 운영진 자원자 2배

배운 것

제품이 아니라 사람의 흐름을 설계하는 일이라는 점에서 PM 일과 비슷했습니다. “왜 흩어지는가” 를 묻는 습관이 RoomSync에서 “왜 겹치는가”로 자연스럽게 이어졌습니다.
