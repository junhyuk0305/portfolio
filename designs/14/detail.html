[01] // 문제 01

50여 동아리,
수기 예약이
겹친다.

학내 50여 개 동아리가 공용 동아리방을 카톡·노트로 예약하면서 이중 예약과 노쇼가 일상이었다. 회장단 설문 결과 가장 큰 페인은 "지금 누가 쓰고 있는지" 실시간으로 알 수 없다는 점이었다.

  • SCALE대상: 학내 동아리 약 50개 · 공용 동아리방 다수
  • PAIN수기(카톡/시트) 예약 → 시간 겹침, 노쇼
  • NEED회장단 설문 1순위 = 실시간 사용 현황
FIG.01 · 회장단 설문 / 현장 관찰
[02] // 리서치 02

예약 흐름을
3가지 시나리오로
정리했다.

회장단·일반 부원 인터뷰를 통해 "예약 → 사용 → 취소 / 노쇼" 흐름과 충돌이 발생하는 시점을 분리해 정의했다. MVP가 반드시 다뤄야 할 기능은 예약, 취소, 실시간 사용 현황 뷰 3가지로 좁혔다.

  • QUAL회장단·부원 인터뷰로 페인 포인트 정리
  • SCOPEMVP 범위: 예약 · 취소 · 실시간 현황
  • SPECAPI 명세 초안 작성 → 개발자와 합의
FIG.02 · MVP 범위 / API 명세 초안
[03] // 실행 03

Jira · 스크럼으로
1개월 만에 출시.

Jira로 백로그를 쪼개고 짧은 스프린트로 개발자 2명과 협업했다. Figma로 핵심 화면만 잡고, "노쇼 감소"라는 단일 성공 지표를 기준으로 기능 우선순위를 단순화해 1개월 만에 정식 배포했다.

  • PRIORITY실시간 현황 뷰 → 예약 → 취소 순서로 출시
  • PROCESSJira 백로그 + 스크럼(스프린트 단위)
  • DESIGNFigma로 핵심 화면 와이어프레임
  • METRIC성공 지표 = 노쇼율 1개
FIG.03 · 스프린트 보드 / 출시 회고
