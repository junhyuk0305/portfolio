CASE STUDY / 01

RoomSync
동아리방 예약 관리 웹서비스

Year
2023.07 – 2023.11
Role
Service Planner · Scrum Master
Team
기획 1 · 백엔드 2
Stack
Notion · Figma · Jira
01 / Context

수기 예약이 만들어내는 끊임없는 겹침 사고

학내 50여 개 동아리가 공용 회의실을 수기로 예약하면서, 겹침 사고가 빈번하게 반복되고 있었습니다. 회장단이 진행한 구글폼 설문을 통해 '지금 이 시각 어느 방이 비어 있는가'를 즉시 확인하지 못하는 정보 비대칭이 문제의 근원임을 확인했습니다.

설문에서 '실시간 예약 현황 확인'이 핵심 니즈로 도출되었고, 이 데이터는 이후 MVP 범위를 결정하는 결정적 근거가 되었습니다.

02 / Approach

가설을 좁히고, 검증 가능한 단위로 쪼개기

'실시간으로 공용 회의실의 사용 현황을 한 화면에서 확인할 수 있다면 노쇼와 겹침이 의미 있게 줄어들 것이다'라는 단일 가설로 수렴시켰습니다. 검증 지표는 노쇼율과 가입 동아리 수 중심으로 한정했습니다.

Notion에 가설과 지표를 매핑해, 회의에서 "이 기능은 어떤 가설을 검증하는가?"가 즉시 답변 가능하도록 설계했습니다. 덕분에 우선순위 논쟁이 감정이 아닌 데이터로 정리되는 흐름이 만들어졌습니다.

03 / Decision

MVP는 예약·취소·실시간 뷰 세 가지로

백엔드 2명에 학기 일정상 시간이 부족한 상황이었습니다. 모든 요구사항을 담는 대신, 가장 큰 페인포인트를 해결하는 핵심 기능 — 예약·취소·실시간 뷰만 MVP에 남기고, 알림은 2차 스프린트로 미루는 결정을 내렸습니다.

리소스 한계를 회장단에 솔직히 공유하고 함께 우선순위를 정리한 결과, 이해관계자 모두가 MVP 범위에 동의한 상태로 개발에 진입할 수 있었습니다. 이는 출시 직전 스코프 크리프를 예방한 결정적인 합의였습니다.

04 / Execution

API 명세 초안을 직접 그려 소통 비용을 줄이다

엔드포인트·요청·응답 스키마의 초안을 기획자가 직접 작성해 백엔드에 전달했습니다. 개발팀은 비즈니스 로직 검증과 데이터 모델 보강에 집중할 수 있었고, '이 화면에서 무엇이 필요한가'를 다시 묻는 미팅이 사실상 사라졌습니다.

Jira로 백로그를 관리하고 스크럼 마스터 역할을 맡아, 스프린트 회고 결과를 다음 백로그에 반영하는 과정을 통해 팀이 스스로 개선하는 리듬을 만들었습니다.

Artefacts: PRD · API Spec 초안 · Sprint Backlog (Jira)
05 / Outcome

출시 30일, 학내 표준이 된 RoomSync

런칭 1개월 만에 학내 30개 동아리가 자발적으로 가입했고, 도입 동아리 기준 노쇼율이 출시 전 대비 60% 감소했습니다. 회장단 정기회의에서 RoomSync가 '학내 표준 예약 도구'로 공식 채택되며 도입 동아리 수가 지속 확대되었습니다.

가장 큰 성과는 숫자 그 자체보다, 한정된 리소스 안에서 '무엇을 만들지 않을 것인가'를 데이터로 결정한 경험이었습니다. 이 의사결정 프레임은 이후의 모든 프로젝트에 그대로 이식되었습니다.

06 / End
