Case 01 — RoomSync

겹치지 않는 시간표는, 그냥 약속이 아니라 시스템의 문제다.

학내 50여 개 동아리가 공용 회의실을 수기로 예약하다 겹침 사고가 반복되던 문제. 회장단 설문으로 핵심 니즈를 검증한 뒤, 예약·취소·실시간 뷰만 남긴 MVP를 5개월 안에 출시했습니다.

Chapter 01 — Discovery
Chapter 01 — 발견

먼저 묻는다, 어디서 막히는지.

회장단 구글 폼 설문으로 검증한 니즈

50여 개 동아리가 공용 회의실을 종이와 단톡방으로 관리하다 보니, 시간 겹침과 노쇼가 반복되고 있었습니다. 가설을 검증하기 위해 회장단 대상 구글 폼 설문을 돌렸고, ‘실시간 예약 현황을 한눈에 볼 수 있어야 한다’는 항목이 압도적인 1순위 니즈로 확인되었습니다.

문제는 곧 ‘기능 욕심’이 아니라 ‘현황 가시화’였습니다. 알림과 통계는 매력적이지만, 현장 운영을 무너뜨리는 진짜 통점이 아니라는 사실을 데이터로 못 박은 뒤 다음 단계로 넘어갔습니다.

  • 대상: 학내 50여 개 동아리 회장단
  • 수단: 구글 폼 설문 — 우선순위 평가
  • 핵심 니즈: 실시간 예약 현황 확인
  • 2차 니즈로 분류: 자동 알림 · 통계 리포트
Chapter 02 — Build
Chapter 02 — 실행

덜어내야 빠르게 도착한다.

백엔드 2명 · 일정 부족 · MVP 결정

백엔드 개발자는 두 명, 학기 일정은 빠듯했습니다. 모든 기능을 욕심내는 대신, 예약·취소·실시간 뷰 세 가지만 1차 스프린트에 포함시키고 알림과 통계는 2차로 미루는 결정을 팀과 합의했습니다.

개발 소통 비용을 줄이기 위해 API 명세서 초안을 직접 작성해 개발팀에 전달했고, Jira를 도입해 스크럼 마스터로서 데일리·회고를 운영하며 일정을 직접 관리했습니다.

  • MVP 범위: 예약 · 취소 · 실시간 뷰
  • 2차 백로그: 알림 · 통계 리포트
  • PM이 직접 작성한 API 명세 초안
  • Jira 도입 · 스크럼 마스터 · 데일리/회고 운영

Chapter 03 — 결과

출시 1개월 시점

‘현황을 보이게 만든다’는 단 하나의 약속을 끝까지 지킨 결과, 출시 1개월 만에 학내 30개 동아리가 자발적으로 가입했고, 가장 큰 통점이었던 노쇼 비율은 이전 운영 대비 60% 감소했습니다. 다음 분기 백로그로는 회장단의 운영 부담을 줄여줄 알림 모듈을 우선 배치했습니다.
