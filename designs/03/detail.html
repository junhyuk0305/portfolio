Case 01 — RoomSync

수기로 굴러가던 동아리방 예약을, 실시간 웹서비스로.

학내 50여 동아리가 공용 회의실을 수기로 예약하다 겹침 사고가 잦았습니다. 회장단 구글폼 설문으로 핵심 니즈를 검증하고, 백엔드 2명·시간 부족 상황을 고려해 MVP 웹서비스로 다시 설계했습니다.

Step 01 — Problem

"방이 비었는지 매번 가서 확인해야 한다."

학내 50여 동아리가 공용 회의실을 수기 노트로 예약하다 보니, 같은 시간대 겹침 사고가 잦았습니다. 노트를 확인하러 학관까지 헛걸음하는 일도 반복되었습니다.

  • 수기 운영 — 같은 시간대 예약 겹침 사고 빈발
  • "지금 비었는지"를 확인할 방법이 노트 외에는 없음
  • 노쇼 시 다음 팀이 알 수 없어 회의실이 그대로 빈 채 방치
Step 02 — Research

회장단 설문으로 '진짜 니즈'를 분리.

회장단 대상 구글폼 설문을 돌렸습니다. "예쁜 UI"나 "추가 기능"보다 "지금 비었는지 알려달라"는 실시간 현황 확인이 압도적인 1순위로 나왔습니다.

"기능은 적어도 됩니다. 가서 헛걸음만 안 하면 돼요."
  • 핵심 니즈 · 실시간 공실 현황 확인
  • 예약·취소 절차의 단순화
  • 모바일에서 즉시 확인 가능한 접근성
정리 — 구글폼 응답을 우선순위로 분리
Step 03 — Decision

기능을 3가지로 줄였습니다.

백엔드 2명·학기 내 출시라는 제약 안에서 "예약 / 취소 / 실시간 뷰"만 v1 범위로 잡았습니다. 알림 기능은 가장 큰 욕심이었지만 2차로 미루기로 합의했습니다.

MVP 우선예약·취소·실시간 뷰알림은 2차
Step 04 — Execution

API 명세부터 스크럼 운영까지.

예약·취소·현황 도메인의 API 명세 초안을 직접 작성해 개발자와 합의 기반을 만들었습니다. Jira 보드로 일감을 트래킹하고, 스크럼 마스터 역할로 짧은 주기 회고를 통해 막힌 지점을 빠르게 풀었습니다.

  • API 명세 초안 — 개발 합의 시간을 단축
  • Jira로 스프린트 운영 — 우선순위 가시화
  • 스크럼 마스터로 회고·데일리 리듬 유지
Step 05 — Result

한 달 안에 30개 동아리, 노쇼 −60%.

출시 한 달 시점에 30개 동아리가 자발적으로 가입했고, 노쇼는 도입 전 대비 60% 감소했습니다. "실시간 현황 확인" 한 가지 가치에 집중한 결정이 짧은 시간 안에 신뢰로 돌아왔습니다.

Step 06 — Reflection

"덜 만드는 것이 더 만드는 것이었다."

가장 어려운 결정은 알림 기능을 v1에서 빼는 일이었습니다. 그러나 핵심 가치인 '실시간 현황 확인'에 집중한 덕분에 짧은 시간 안에 사용자의 신뢰를 얻을 수 있었습니다.

다음 프로덕트에서도 같은 태도를 유지하려 합니다 — 가설은 설문으로, 결정은 제약 안에서, 실행은 팀과 함께.
