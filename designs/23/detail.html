★ CASE 01 — RoomSync ★

학내 동아리방 예약,
겹침을 0으로.

50여 동아리가 수기로 관리하던 동아리방 예약 시스템을 회장단 설문으로 문제를 정의하고, MVP로 한 달 만에 검증한 학내 웹서비스 기획·런칭 케이스.

2023.07 — 11 서비스 기획 · PM Jira · 스크럼 MVP 런칭
★ TEAM @ LIBRARY · SPRINT 01
01.PROBLEM · 무엇이 문제였나

수기 예약 + 실시간 현황 부재

학내 50여 개 동아리가 단톡방·구글시트로 동아리방을 수기 예약하다 보니 겹침과 노쇼가 반복됐습니다. 회장단을 대상으로 설문을 돌린 결과, 가장 강한 니즈는 "지금 누가 어디를 쓰고 있는지" — 실시간 현황이었습니다.

  • 50여 동아리, 단일 예약 채널 없음
  • 겹침·노쇼 발생 시 책임 소재 불분명
  • 회장단 설문 1순위 니즈: 실시간 점유 현황
02.RESEARCH · 리서치

회장단 설문으로 가설을 좁히다

화려한 기능보다, 회장들이 매주 마주치는 진짜 통증을 묻는 것이 먼저였습니다. 설문 응답을 정리해 "예약 / 취소 / 실시간 뷰" 3가지로 MVP 범위를 잘랐고, 운영 동선을 직접 따라가며 흐름을 검증했습니다.

  • 핵심 가설: 실시간 점유 뷰만 있어도 노쇼가 줄 것
  • 범위 컷: 예약 · 취소 · 실시간 뷰 (Out: 결제, 멤버 관리)
  • API 명세 초안 작성 후 개발팀과 합의
★ INTERVIEW · 회장단
03.EXECUTION · 실행
★ SPRINT · PM @ DESK

Jira · 스크럼으로 한 달 안에 MVP

API 명세 초안을 기반으로 백엔드·프론트와 화면 흐름을 합의하고, Jira로 스프린트를 관리했습니다. 매주 회장단 1팀에게 빌드를 보여주며 흐름을 조정했고, 한 달 만에 30개 동아리에 배포할 수 있는 수준으로 다듬었습니다.

WEEK 01설문·문제 정의, MVP 범위 확정
WEEK 02API 명세 초안, 화면 흐름 합의
WEEK 03예약/취소/실시간 뷰 구현 · 내부 테스트
WEEK 0430개 동아리 대상 정식 배포
04.RESULT · 결과

실시간 현황 한 화면이 노쇼를 줄였다

실시간 점유 뷰가 가시화되자 회장들이 직접 예약을 정리·취소하기 시작했고, 노쇼가 눈에 띄게 줄었습니다. 화려한 기능 없이 "현재 상태"를 보여주는 것만으로 운영이 정돈된 케이스였습니다.

05.LEARNING · 배움

"무엇을 빼느냐"가 PM의 일

처음에는 결제·멤버 관리까지 욕심을 냈지만, 회장단 설문이 말한 핵심은 단 하나 — 실시간 뷰였습니다. 한 달 안에 검증하기 위해 무엇을 빼느냐가 PM의 진짜 일이라는 것을 체감했습니다. 이 경험은 이후 A사 인턴십에서 5명 UT로 가설을 좁혀 +12%p 전환을 만든 작업으로 이어졌습니다.
