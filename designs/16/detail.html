CASE STUDY · 01

RoomSync 🚪
동아리방 예약 서비스

Role

PM · 스크럼 마스터

Period

2023.07 — 2023.11 (1개월 운영)

Scope

학내 50여 동아리

Stack

Figma(기본) · Jira · Notion

01 / PROBLEM

문제 정의

학내 50여 개 동아리, 공용 회의실 수기 예약의 한계

학내 50여 개 동아리가 공용 회의실을 노션·카톡으로 수기 예약하며 예약 겹침과 노쇼가 반복되고 있었습니다. 회장단과 대화하며 "다른 동아리가 이미 쓰고 있는지 알 수 없다"가 가장 큰 불편으로 반복해 등장하는 것을 확인했고, 저는 이 비효율을 해결하고자 직접 팀을 꾸려 PM 역할을 자청했습니다.

  • 예약 겹침으로 인한 회장단 간 마찰
  • 예약 후 무단 불참(노쇼)으로 인한 빈 공간 발생
  • 예약 채널 분산 (단톡방·노션·수기 시트)
02 / RESEARCH

리서치 — 데이터로 핵심 니즈를 검증

회장단 설문으로 '실시간 현황' 니즈 검증

가설을 검증하기 위해 학내 동아리 회장단을 대상으로 설문을 진행했습니다. 가장 원하는 기능으로 '실시간 예약 현황 확인'이 1순위로 도출되어, 이 한 가지 가치를 중심으로 MVP의 범위를 좁히기로 결정했습니다.

회장단 설문 · 정성 페인포인트 수집
  • 1순위 니즈: 실시간 예약 현황 확인
  • 2순위 니즈: 모바일에서의 즉시 예약
  • 핵심 인사이트: '겹침 사고'와 '노쇼'가 동시에 해결되어야 함
03 / EXECUTION

실행 — MVP 전략과 스프린트 운영

핵심 3기능에 집중, 나머지는 다음 단계로

제한된 리소스 안에서 한 학기 내 런칭이 목표였습니다. 핵심 기능(예약·취소·실시간 뷰)만 MVP로 남기고, 알림·통계 등 부가 기능은 과감히 후순위로 미루는 전략을 택했습니다. 복수전공으로 쌓은 개발 지식을 살려 API 명세서 초안을 직접 작성했고, Jira·스크럼으로 개발팀과 호흡을 맞췄습니다.

Jira·스크럼 기반 스프린트 운영
  • API 명세서 초안 직접 작성 → 개발팀과의 R&R 명확화
  • Jira·스크럼으로 백로그·스프린트 관리
  • MVP 원칙으로 범위 통제 (예약 / 취소 / 실시간 뷰)
04 / RESULT

결과 — 한 달 만에 30개 동아리, 노쇼율 60% 감소

출시 후 한 달, 의미 있는 정량 지표

출시 후 한 달 만에 학내 30개 동아리가 자발적으로 가입했고, 노쇼(No-show) 비율이 기존 대비 60% 감소했습니다. 실시간 뷰 도입만으로도 회장단 간 겹침 사고에 대한 마찰이 눈에 띄게 줄었습니다.

05 / LEARNING

회고

PM이 직접 API 명세를 쓸 수 있다는 것의 의미

이 프로젝트는 비즈니스의 니즈를 데이터로 검증하고, 개발팀과의 인터페이스를 PM이 직접 좁힐 수 있을 때 실행 속도가 얼마나 달라지는지를 체감한 계기였습니다. '기획자는 무엇을 만들지 정하고, 개발자는 어떻게 만들지 정한다'는 분리된 사고가 아니라 '어떻게'까지 같이 고민해야 진짜 빠른 프로덕트가 나온다는 것을 배웠습니다.
