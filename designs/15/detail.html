🎯 문제 정의

학내 50여 개 동아리가 공용 동아리방을 수기로 예약하는 환경이었습니다. 중복 예약과 노쇼가 일상이 되며 룸을 정말 필요로 하는 팀의 미팅이 취소되는 악순환이 반복되었습니다.

중복 예약 노쇼 수기 운영
🔍 리서치

회장단을 대상으로 설문을 진행해 핵심 니즈를 데이터로 검증했습니다. 가장 빈번하게 등장한 키워드는 ‘실시간 현황’이었습니다.

  • “실시간 예약 현황 확인”이 가장 빈번한 1순위 니즈로 지목
  • 알림·리뷰 같은 부가 기능은 후순위로 분류
  • “예약 전에 칠판 확인하러 가는 동선이 불편하다”는 응답 다수
회장단 설문 니즈 우선순위화
🔍 회장단 설문 정리
⚖️ MVP 결정

리소스 제약 안에서 “있으면 좋은 것”을 과감히 자르기로 결정했습니다.

  • 1차 (MVP) — 예약 · 취소 · 실시간 룸 뷰만 포함
  • 2차 — 알림 기능으로 확장
  • 의사결정 근거: 리서치에서 검증된 ‘실시간 현황’이 단일 핵심 가치
MoSCoW 스프린트 분할 의사결정 로그
🛠 실행과 협업

개발팀과의 소통 비용을 직접 줄이기 위해, 기획자가 먼저 API 명세서 초안을 작성해 핸드오프했습니다. 이후 백엔드 리뷰를 거쳐 확정하는 방식으로 회의 시간을 절반으로 줄였습니다.

  • Jira 도입 · 스크럼 마스터로 데일리 스탠드업 운영
  • Figma로 와이어프레임 → 인터랙션 흐름까지 직접 작성
  • 주간 회고에서 “기획자가 명세 미리 그려와서 좋다”는 피드백
API 명세 초안 Jira · 스크럼 Figma 와이어
🛠 API 명세 직접 작성 · 핸드오프 문서
📈 결과

출시 첫 달, 학내 동아리들의 입소문으로 30개 팀이 RoomSync에 가입했고, 수기 운영 대비 노쇼율이 60% 감소했습니다.
