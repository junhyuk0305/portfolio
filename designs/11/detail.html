문제 정의
CONTEXT

수기 예약이 만든 겹침 사고

학내 50여 개 동아리가 공용 회의실 몇 곳을 단톡방과 종이 노트로 예약하다 보니, 같은 시간에 두 동아리가 들이닥치는 겹침 사고가 반복됐습니다.

  • 예약 정보가 단톡·메모·구글캘린더로 분산
  • "누가 언제 잡았는지" 회장단도 실시간으로 모름
  • 노쇼·취소 통지가 다음 사용자에게 전달되지 않음

회장단 설문에서 본 1순위 니즈

회장단을 대상으로 구글폼 설문을 돌린 결과, "실시간 예약 현황 확인"이 압도적인 1순위 니즈로 나왔습니다. 거창한 매칭이 아니라 "지금 비어있는지"가 문제였습니다.

  • 1순위 — 실시간 예약 현황 한눈에 보기
  • 2순위 — 예약·취소를 한 화면에서 처리
  • 3순위 — 노쇼 방지를 위한 리마인드 알림
// 002
스코프 결정
SCOPE
SURVEY · 회장단 N

백엔드 2명 · 학기 일정 안에 끝낸다

백엔드 인력 2명과 학기 일정이라는 현실 제약을 먼저 인정했습니다. 알림·통계는 2차로 미루고, MVP는 "예약·취소·실시간 뷰" 세 가지에만 집중했습니다.

  • MVP — 예약 생성, 취소, 실시간 캘린더 뷰
  • 2차 — 리마인드 알림, 노쇼 카운트
  • API 명세 초안을 직접 작성해 BE/FE 핸드오프 단축
  • Jira에서 스크럼 마스터로 2주 스프린트 운영
// 003
실행
EXECUTION

한 학기 안에 30개 동아리까지

MVP 배포 후 회장단 채널을 통해 동아리들에게 순차 온보딩했고, 1개월 만에 30개 동아리가 실사용에 진입했습니다.

  • 회장단 6명 베타 → 핵심 플로우 검증 후 확산
  • 이용 가이드 PDF 1장 + 단톡 안내 템플릿 배포
  • 주간 사용 로그를 회장단에 공유, 불편 리포트 회수
  • 알림(2차) — 예약 30분 전 리마인드로 노쇼 대응
