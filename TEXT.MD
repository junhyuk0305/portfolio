# 🧭 3. 핵심 설계 원칙 (강사용 메모)

본 강의안은 **"원하는 디자인을 손에 든다 / 25종 검증 갤러리가 디테일을 보강한다 / 자소서가 정합성을 검증한다"** 라는 3축 원칙으로 설계되었습니다.

- **핵심 강의 목표**: 단순히 "포트폴리오 사이트 1개"가 아니라, "**본인이 원하는 디자인의 인터랙티브 웹 포트폴리오**"를 손에 들고 나가는 것.
- 자소서에서 뽑는 브랜드 키는 7블록 — 페르소나 + 무드 + Hero 카피 + 7축 비주얼 디렉션 카드. 이 카드가 다음 단계의 정합성 기준이 됩니다.
- 디자인 출처는 두 곳에서 합성합니다: (A) 학생이 awwwards/godly/siteinspire 등에서 직접 캡쳐한 1장, (B) 강사가 제공한 25종 검증 갤러리. AI가 둘을 합성해 "디자인 디테일 카드 1장"으로 떨어뜨립니다.
- 컬러 Hex, 폰트 페어링, 레이아웃, 인터랙션은 이 디자인 디테일 카드가 100% 결정 — 학생은 디자인 디렉터 역할에서 해방되되, 결과물의 "원하는 디자인" 정체성은 본인이 고른 캡쳐가 책임집니다.
- 인지 부하 ↓ / 결과물 격 ↑ / "내가 만든 것 같다"는 주관적 만족 ↑ / 강의 시간 안에 마감 가능.

본 강의안의 모든 예시 페르소나는 **김태윤 (PM, person.md)** 으로 통일되어 있습니다. 한 사람의 흐름을 끝까지 따라가며 학생의 학습 부담을 줄입니다.

## SLIDE 10 — 500자 자소서 = 포트폴리오 UIUX의 '정합성 필터'

**[화면 구성]**
- 중앙: 자소서 카드 → 화살표 → 우측에 결과물 3종
  - 무드 키워드 3개 (예: 분석적인 / 신뢰감 / 구조적인)
  - Hero 카피 (예: "병목을 푸는 기획자")
  - 7축 비주얼 디렉션 카드 (패밀리·컬러·타이포·밀도·모션·이미지·첫인상)
- 하단 가로 다이어그램: [본인이 원하는 레퍼런스 캡쳐] ←→ [자소서가 정의하는 7축 카드] = "정합성 체크"
- 하단 카피: "디자인 = 본인 욕망(캡쳐) × 자소서 정합성. 둘이 어긋나면 첫인상도 어긋난다."

**[강사 멘트]**
> "여기서 가장 중요한 관점 하나를 가져가셔야 합니다.
>
> 500자 자소서는 단순한 글이 아니라,
> 본인이 고른 '원하는 디자인'이 본인 정체성과 맞는지 검증하는 '정합성 필터'입니다.
>
> 김태윤처럼 '데이터로 증명한다, 침착하게 분석한다' 톤인 사람이
> 만약 컬러풀하고 톡톡 튀는 바이럴 사이트를 캡쳐해 가져왔다고 칩시다.
> 그럼 첫인상에서 사람과 사이트가 다른 두 명처럼 보여요.
> 그래서 자소서가 뽑아낸 '무드 + 7축 비주얼 디렉션 카드'가
> 본인이 가져온 캡쳐와 한 번 합을 맞춰주는 거예요.
>
> 디자인을 고르는 건 본인 욕망에서 시작합니다.
> 다만 그 욕망이 본인 자소서와 한 사람으로 보이는지, 자소서가 검증해줍니다.
> 두 신호가 맞아야 첫인상이 강해집니다."

**[전환 포인트]** "그럼 이 정합성 체크를 사람 손이 아니라 AI에게 시켜보겠습니다."

## SLIDE 18 — STAR 프레임워크 정의

**[화면 구성]**
- 4분할 박스
  - 🟦 **S — Situation (상황)**: 이 프로젝트가 시작된 시점의 시장/조직/사용자 맥락
  - 🟧 **T — Task (해결할 과제)**: 그 상황에서 내가 풀어야 했던 명확한 문제
  - 🟩 **A — Action (나의 행동)**: 내가 실제로 한 일과 사용한 방법론·툴
  - 🟥 **R — Result (성과)**: 정량적 지표 + 정성적 변화
- 하단 카피: "면접관·인사담당자가 가장 빠르게 후보자를 평가하는 4문항"

**[강사 멘트]**
> "STAR는 면접관과 인사담당자가 한 사람의 경험을 평가할 때
> 머릿속에서 자동으로 던지는 4가지 질문입니다.
> '어떤 상황이었는데, 뭐가 문제였고, 너는 뭘 했고, 결과가 어땠어?'
>
> 그래서 STAR로 정리된 텍스트는 그 자체로 인사담당자의 사고 순서와 일치합니다.
> 우리는 여기서 한 발 더 갑니다.
> STAR로 정리된 경험을 곧바로 포트폴리오 UI 블록으로 치환합니다."

**[전환 포인트]** "김태윤 RoomSync 프로젝트가 어떻게 STAR로 정리되는지 보여드리겠습니다."

## SLIDE 37 — [실습 ③] GitHub Pages Step-by-Step (15분)

**[화면 구성]**
- 5단계 스크린샷 슬라이드 (좌→우 흐름)

- `https://github.com` 접속 → Sign up
- ⚠️ 사용자명이 곧 URL이 됩니다 (`https://(사용자명).github.io`)
- 김태윤 예시: `taeyoonkim-portfolio` → 최종 URL `https://taeyoonkim-portfolio.github.io`

**Step 2. New Repository 생성**
- 우측 상단 + 버튼 → New repository
- Repository name: **`(사용자명).github.io`** ← 정확히 이 규칙으로 입력 필수
- Public 체크 / Add a README file 체크 → Create repository

**Step 3. index.html 업로드**
- 생성된 Repository 화면에서 Add file → Upload files
- Gemini AI Studio에서 받은 index.html을 드래그앤드롭
- 하단 Commit message는 비워둔 상태로 Commit changes 클릭

**Step 4. Pages 활성화**
- 상단 메뉴 Settings → 좌측 사이드바 Pages
- Source: Deploy from a branch
- Branch: `main` / 폴더: `/ (root)` → Save
- 상단에 노란색 "Your site is being built" 메시지 → 1~2분 대기

**Step 5. 라이브 확인**
- 동일 Settings → Pages 화면에서 초록색 박스로 URL 표시
- 클릭 → 본인 사이트가 진짜로 열림
- ⚠️ 첫 빌드는 시간이 좀 걸립니다 (404 뜨면 1~2분 더 대기 후 새로고침)
